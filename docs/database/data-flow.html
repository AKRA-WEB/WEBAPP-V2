App 1

Main / SSO / Core Admin

Implemented RLS

Core เป็นฐานร่วมของทุกแอพ: login, profile, role, permission, app registry, และ audit trail. V1 core import นำผู้ใช้/roles/permissions เข้าสตaging แล้ว.

  1. V1 core snapshots

    User, RoleConfig, PermConfig ใช้เป็น reference/import source.

  2. Supabase Auth

    สร้าง auth.users; trigger private.handle_new_user() สร้าง profiles.

  3. Role mapping

    profiles -> user_roles -> roles.

  4. Permission mapping

    roles -> role_permissions -> permissions.

  5. App registry

    apps.required_permission กำหนดว่าหน้า Main แสดง/ล็อก module ใด.

  6. Guards and RLS

    requirePermission() และ private.has_permission() ใช้ enforce สิทธิ์.

Primary tables
profiles, roles, permissions, user_roles, role_permissions, apps, audit_logs
App routes
/, /login, /admin/permissions
Key rule
ห้าม authorize จาก user-editable metadata; permission ต้องมาจาก DB-backed roles/permissions.

    Normalize product code/name, vendor, warehouse, location, stock snapshot, movement history.

  3. Canonical product

    catalog_products stores shared identity and default display name.

  4. Source aliases

    catalog_product_aliases preserves app-specific product names and match status.

  5. Scopes

    catalog_product_scopes answers TRD/AKRA/W5/module membership.

  6. Consumers

    Picking create uses matched aliases now; future modules reuse the same catalog.

App 2

Picking

List/create/status implemented Problem flow local Server-only writes

Picking เป็น pilot ที่มี data flow สมบูรณ์ที่สุดในตอนนี้: read-only list/detail, create requisition, status transition และ local problem-reporting flow.

Create requisition flow

  1. /picking/new

    Writer/admin เลือก staff, bill type, และ product alias หรือ free text.

  2. Server action

    createPickingRequisition() checks picking.write.

  3. Service-role RPC

    public.create_picking_requisition() runs as service role only.

  4. Bill number

    private.next_picking_bill_no() updates picking_daily_sequences.

  5. Write rows

    Insert picking_requisitions, picking_requisition_lines, and created event.

Status and problem flow

  1. Detail page

    /picking/[id] reads requisition, lines, events, and problem reports through RLS.

  2. Status action

    transition_picking_requisition_status() allows only pending -> picked -> sent.

  3. Problem action

    report_picking_problem() writes report + lines + event, status unchanged.

  4. Review

    Detail page shows lifecycle and problem report history for readers/writers.

Primary tables
picking_requisitions, picking_requisition_lines, picking_requisition_events, picking_problem_reports, picking_problem_report_lines
Reference/secret tables
picking_staff, picking_staff_line_accounts, picking_requisition_secrets, picking_daily_sequences
Catalog bridge
picking_requisition_lines.catalog_product_id -> catalog_products.id; catalog_alias_id -> catalog_product_aliases.id
Next confirmed dependency
LINE notification/failure recovery should start with disabled send/dry-run behavior.

Cross-cutting

Notifications / LINE

Planned Server-only

Notifications เป็น cross-cutting app/service ไม่ใช่ตาราง workflow หลักในตอนนี้. การส่ง LINE ต้องทำฝั่ง server เท่านั้น และ stage แรกต้องเป็น disabled send/dry-run.

  1. Workflow event

    Picking/PO/GR/returns writes an event that may require notification.

  2. Server job/action

    Server code reads secret/contact data and builds payload.

  3. Dry-run result

    First phase records intended send/result without calling external LINE.

  4. Failure recovery

    Later flow records failure state and supports retry/recovery.

Current Picking secret/contact tables

Future table family

Key rule
